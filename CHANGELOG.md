- Git parity: `Version.Diff` gained `--name-only`/`--name-status` (changed paths, optionally prefixed with git's A/D/M status), and `Version.Log.Log_Formatted_From_Commit` renders a first-parent walk through `Version.Pretty_Format` (git's `--format`/`--pretty=format:`/`tformat:`, with the right per-record vs separator newline).
- Git parity: `Version.Pretty_Format` `%b` and `%B` now end with git's single trailing newline (`%b` is empty with no newline when there is no body), instead of dropping it. `Version.Notes.Add` now normalises a note message the way git does — trailing blank lines/whitespace stripped and exactly one trailing newline ensured — so a version-written note object is byte-identical to git's (same SHA), and `version notes show` emits the note verbatim (via `Version.Console.Put`) rather than doubling its final newline.
- Bug fix: `Version.Upload_Pack.Build_Want_Request` (filtered fetch/partial clone) now advertises the `filter` capability in the want line. Without it, upload-pack rejected the following `filter <spec>` command and returned no usable pack, so `clone --filter=...`/`fetch --filter=...` over a filter-enabled HTTP/SSH server failed with "upload-pack response did not provide requested commit". Filtered clones now complete and are `git fsck`-clean.
- Bug fix: object reachability (`Version.Reachability.Repository_Roots`, used by `verify`/`gc`) is now reftable-aware — it enumerates branch/tag/remote roots from the reftable stack rather than walking `.git/refs/heads` etc. A git-created reftable repository leaves those as stub files, which previously crashed the loose-ref walk with `unknown directory "heads"`. The loose/reflog directory walkers also now skip a non-directory path defensively.
- Git parity: `Version.Push.Delete_Ref` (`push --delete`, `push :REF`) now resolves an unqualified ref name against the remote's advertised refs — preferring a matching branch, then a tag — instead of always assuming `refs/heads/`. Deleting a short tag name (e.g. `push --delete origin v1`) now removes `refs/tags/v1` like git, across local, HTTP, and SSH remotes.
- Git parity: `Version.Describe.Describe` now considers only annotated tags by default (git's rule) and gains an `All_Tags` argument (the CLI's `describe --tags`) that also considers lightweight tags. Its failure messages now mirror git: "No names found, cannot describe anything." when no tag is reachable, and "No annotated tags can describe '<oid>'." with the "However, there were unannotated tags: try --tags." hint when only lightweight tags are reachable. Previously a lightweight tag was silently used by default.
- Git parity: `Version.Push.Push_Default` (a `push`/`version push REMOTE` with no refspec) now falls back to `push.default` when no `remote.<name>.push` is configured, instead of erroring. It implements git's `simple` (default), `current`, `upstream`/`tracking`, and `nothing` modes — pushing the current branch to its same-named or upstream branch — and reproduces git's errors for a missing upstream, a `simple` name mismatch, `nothing`, and a detached HEAD. (`matching` continues to route to `Push_Matching`.)
- Git parity: `Version.Write.Save`/`Save_Amend` now honour commit signing — they read `commit.gpgSign` (with the key from `user.signingKey`, else gpg's default) and gain a `Sign`/`Signing_Key` parameter so the CLI's `-S`/`--gpg-sign[=KEY]`/`--no-gpg-sign` can force or suppress signing. Previously a configured `commit.gpgSign` (or `save -S`) silently produced an unsigned commit. Config reads are guarded with `Has_Key` so an unconfigured `user.signingKey`/`commit.gpgSign` no longer raises "config key does not exist" from every unsigned commit.
- Bug fix: `Version.Rebase.Start_Interactive` now aborts an emptied/all-comment todo with "nothing to do" and leaves the branch untouched (git's safety net), instead of resetting the branch to the upstream and silently discarding every commit. An explicit `drop` of every commit is still a successful rebase that advances to the upstream — the two are distinguished by whether any command line was seen.
- Git parity: `Version.Clean.Candidates` with `-d` now also removes untracked directories that hold no untracked file (an empty directory, or a nested tree of empty directories collapsed to its shallowest member) by walking the worktree — these contribute no entry to the untracked-file scan. An ignored empty directory still needs `-x`, and symlinks are treated as files (never descended), matching git.
- Windows parity: `Version.Files.To_Native_Path` now applies git-for-Windows's `\\?\` extended-length prefix (and `\\?\UNC\` for UNC paths) to a fully-qualified Windows path of at least MAX_PATH (260) characters, so long absolute paths remain usable via the Win32 API. Relative and short paths are unchanged, and the transformation is a no-op on POSIX. (Runtime behavior on a real Windows host is not part of the Linux test matrix; the path transformation is unit-tested by forcing the platform to Windows.)
- Git parity: `Version.Submodules` gains `Foreach`, `Sync`, and `Deinit` (git's `submodule foreach`/`sync`/`deinit`), operating over the existing `.gitmodules`/gitlink machinery and writing `submodule.<name>.*` config via `Version.Config`. `Version.Working_Tree`'s untracked scan now excludes a `.git` gitfile (not only a `.git` directory), so a submodule/linked-worktree `.git` pointer is no longer reported as untracked.
- Git parity: shallow deepening. `Version.Fetch.Fetch_Deepen` (`fetch --deepen N`) and `Version.Fetch.Fetch_Unshallow` (`fetch --unshallow`) extend or flatten a shallow repository; `Version.Upload_Pack.Build_Want_Request` gained `Relative` (emits the `deepen-relative` capability) and `Have_Shallow` (echoes the client's current boundary as `shallow <oid>` lines), and unshallow uses `deepen 2147483647`. The existing `shallow`/`unshallow` response handling then updates (and removes) `.git/shallow`.
- Bug fix: `Version.Pack.Index_Pack` now finalizes a fetched pack to git's canonical `pack-<checksum>.{pack,idx}` name rather than leaving it at the caller's temporary path, so repeated fetches/clones into one repository no longer collide on a fixed filename and destroy earlier objects.
- Git parity: cone-mode sparse-checkout is now writable. `Version.Sparse.Set_Cone` writes git's cone patterns and sets `core.sparseCheckoutCone`; `Cone_Recursive_Directories` reports the leaf directories for `list`; `Disable` keeps the pattern file (only clearing the config flags); and the cone read logic (`Included`) now honours the `!/dir/*/` ancestor exclusions so a parent directory's own files are kept while its sibling subtrees are excluded. `Version.Staging.Index_Entry` gained a `Skip_Worktree` flag that `Load`/`Write` round-trip (writing a version-3 index with the extended-flags word when any entry sets it), and `Version.Restore.Apply_Sparse_Skip_Worktree`/`Clear_Skip_Worktree` set or clear those bits from the sparse pattern set. `Version.Files.Is_Ordinary_File` returns `False` for an empty path instead of raising.
- Git parity: `Version.Clone.Clone` now tolerates a dangling remote HEAD (the remote's HEAD names a branch that was not fetched): instead of raising, it points HEAD at the unborn default branch, records its `[branch]` upstream config, leaves the worktree unchecked-out and writes no `origin/HEAD`, and warns `remote HEAD refers to nonexistent ref, unable to checkout` — matching `git clone`.
- Git parity: `Version.Config.Read_All` now reads git's full config scope stack — system (`/etc/gitconfig` or `GIT_CONFIG_SYSTEM`, honouring `GIT_CONFIG_NOSYSTEM`), global (`$XDG_CONFIG_HOME/git/config` then `~/.gitconfig`, or `GIT_CONFIG_GLOBAL` replacing both), then the local and per-worktree config — rather than the local config alone, so `Get_Value`/list/keys resolve values defined in the developer's global or system config. `Version.Files.Is_Ordinary_File` now treats an empty path as "not a file" (returns `False` rather than raising `Name_Error`) so suppressed scopes probe cleanly. (`GIT_CONFIG_COUNT` env injection is not yet read.)
- Git parity: `Version.Config.Read_All` now follows `[include]`/`[includeIf]` directives (unconditional plus `gitdir:`, `gitdir/i:`, `onbranch:`, and `hasconfig:remote.*.url:` conditions), resolving include paths relative to the including file with `~` expansion and bounding recursion depth; the directive stays a readable entry as git does. `Get_Value` now returns the **last** matching entry (git's single-value semantics) and `config.worktree` is layered after the common config to match; `Config_Entry_Name` lower-cases the section and variable names (subsection preserved) like git.
- Git parity: `Version.Pathspec` now supports the `:(icase)` magic — a matching item folds ASCII case for the pattern and path while keeping the real path for attribute checks.
- Git parity: SHA-256 object-format repositories are now fully supported, including over the smart HTTP/SSH transport. `extensions.objectFormat = sha256` repositories open (the compatibility gate is lifted), `Version.Init.Init`/`Init_Bare` take an object-format argument, and every object/index/pack/ref/reflog code path is hash-width-aware. The smart transport negotiates the object format (`Version.Upload_Pack.Advertised_Object_Format` reads the advertised `object-format` capability, `Version.Fetch.Remote_Object_Format` discovers it before a clone creates its target, and the receive-pack client sends `object-format=sha256` and repo-width null ids for ref create/delete). Verified against system git — `git fsck`, `verify-pack`, `bundle verify`, `reflog`, and clone/push/fetch over both local and real `git http-backend` SHA-256 servers accept version's output, and version reads git-created SHA-256 repositories (Phases 4–5 of SHA-256 support).
- Added a SHA-256 implementation to `Version.Hash` (`Sha256_Hex`/`Sha256_Raw` plus a streaming `Sha256_Context`) and a `Hash_Algorithm` descriptor (`Hex_Length`/`Raw_Length` = 40/20 for SHA-1, 64/32 for SHA-256) — Phase 0 of SHA-256 object-format support (see `../version/docs/SHA256_SCOPE.md`); no object-format behavior change yet.
- Added `Version.Clone.Clone_Filtered` and `Version.Fetch.Fetch (Remote, Filter_Spec)` for partial clone: HTTP/SSH send the filter when the server advertises the `filter` capability; local transport selectively copies non-blob (or size-bounded) objects via the new `Version.Write.Copy_Object`. The clone is configured as a partial clone, and the local promisor lazy fetch (`Fetch_Object`) now copies only the requested object instead of the whole store.
- Supported the `extensions.worktreeConfig` repository extension: `Version.Config.Read_All` now layers the per-worktree `config.worktree` over the common config (per-worktree values override; absent keys fall back), and `Version.Repository_Format` no longer rejects it.
- Added `Version.Push.Push_Default` (apply the configured `remote.<name>.push` refspec(s) — multi-valued, parsed like command-line refspecs incl. `+`/`:` — via `Version.Config.Read_All`).
- Added a `Force` parameter to `Version.Push.Push_Tags` and `Version.Receive_Pack.Push_Tag`/`Push_Tag_Ssh` (and the local tag-write path) that overwrites a differing remote tag instead of refusing.
- Added `Version.Push.Push_Refspec` and `Version.Receive_Pack.Push_Ref`/`Push_Ref_Ssh` (push an arbitrary source commit to a named remote ref over local, HTTP, or SSH, with a fast-forward check unless forced) — the engine behind `push SRC:DST`.
- Added `Version.Push.Delete_Ref` and `Version.Receive_Pack.Delete_Ref`/`Delete_Ref_Ssh` (delete a remote ref over local, HTTP, or SSH via a delete-only receive-pack command); relaxed `Build_Update_Command` to accept `refs/tags/` (fixing tag push over HTTP/SSH).
- Added a `Force` parameter to `Version.Push.Push` and `Version.Receive_Pack.Push_Branch`/`Push_Branch_Ssh` (and the local branch path) that skips the fast-forward (ancestor) check for non-fast-forward updates.
- Added `Version.Receive_Pack.Push_Tag` / `Push_Tag_Ssh` (push `refs/tags/*` over smart HTTP / SSH receive-pack, refusing to overwrite a differing remote tag); `Version.Push.Push_Tags` now uses them for HTTP/SSH remotes instead of rejecting them, and removed `Version.Unsupported.Push_Tags_Non_Local_Remote`.
- Added `Version.Objects.Tree_Entries` (single-level, non-recursive tree parse returning subtrees as `Tree_Directory` entries) and `Version.Refs.Write_Symbolic_HEAD` (atomically point HEAD at a symbolic ref).
- Replaced the system-`ssh` subprocess SSH transport with native `ssh_lib`: `Version.Transport.Ssh` now opens an `SSH_Lib.Sessions` session and runs `git-upload-pack`/`git-receive-pack`/`git-lfs-authenticate` over `SSH_Lib.Channels.Open_Exec` (streaming via `Read_Some`/`Write`), with no dependency on an external `ssh` binary. Connection settings are resolved through the user's `~/.ssh/config` via `SSH_Lib.Config.Resolve_Remote` (HostName aliases, Port, User, IdentityFile, known_hosts), with the URL's explicit user/port taking precedence. The pure URL/command-builder API is unchanged.

- Added `Version.Rebase.Start_Interactive` (sequence-editor todo): pick/drop/reorder replay through the existing state machine; squash/fixup combine commits via a one-shot executor (using `Write_Commit_With_Author`).
